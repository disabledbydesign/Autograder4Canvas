# Session 2026-04-04 — Design Observations

## Session Stats
- Messages: 11 sent, 0 skipped, 0 flagged
- Admin actions: 26 modules published (2 courses), 35 deadlines pushed (1 course), 2 announcements posted, 1 grade fixed
- Steering instructions: ~6 across 3 conversations (Guiexhuba, ETHN-27BN announcement, Ava Connor)
- Duration: ~2 hours (first-ever session, included tool debugging)

## What Teachers Need (generalizable patterns)

- **Pattern scan before individual processing is essential.** 11 messages looked like 11 separate problems. Grouped, they revealed 2 systemic issues (unpublished modules across courses, broken discussion board) affecting 6 of the 11 messages. An instructor processing sequentially would have responded to the first student's access complaint without discovering the root cause. The tool must default to scan-then-group.

- **Investigation is a core workflow, not an edge case.** Over half the session was spent diagnosing Canvas state — checking module publish status, examining individual submissions, verifying deadline changes landed. Teachers need to understand what's happening on Canvas before they can respond accurately. Draft-then-send is the minority activity.

- **Bulk admin actions must be single-instruction.** "Publish all unpublished modules" and "push all deadlines back one week" each affected dozens of Canvas objects. An instructor doing this manually through the Canvas UI would spend an hour clicking through module settings. One instruction, one confirmation, done. This is the highest-leverage capability.

- **Cross-course awareness surfaces systemic issues.** The same unpublished-module problem appeared in ETHN-27AN and ETHN-27BN independently. Without cross-course grouping, each would look like a course-specific glitch. The tool should detect when the same issue appears across multiple courses.

- **Announcements follow naturally from systemic fixes.** After publishing modules and pushing deadlines, the immediate next step was always "tell the class." The tool should suggest announcements proactively after bulk admin actions. Different courses need different announcement tones for the same underlying fix (online students didn't know about the illness; in-person students did).

- **Prior message search is a reference need.** The instructor said "look at the email I sent Asher for reference" — needing to find a specific prior conversation to inform the current draft. This is a retrieval problem: the tool needs to index sent messages and make them searchable by student name, topic, or date. This came up naturally, not as a pre-planned feature.

- **Session context shapes every draft.** "I've been sick. And I'm losing my job." — shared at session start, applied to every subsequent draft and action. Without this context, drafts would be tone-deaf. This is instructor state, not student state, and it changes infrequently but matters enormously.

## System Adaptation Points

- **Steering reveals over-explanation.** Multiple steering instructions were about removing information students already knew: "they already know I've been sick, no need to re-state it," "don't say bumpy few weeks, they already know." The draft generator defaults to over-explaining. The system should learn: when context indicates students are already aware of something, don't restate it.

- **Steering reveals missing policy details.** "Remind her the minimum grade is a C" — a course policy the context engine had access to but didn't surface in the draft. The system should learn which policies are relevant to which message categories (absence/catch-up → minimum grade matters).

- **Steering reveals voice calibration.** "Not that it won't help her. It's that it will stress her out." — the draft used pragmatic framing ("won't help") when the instructor wanted care-centered framing ("will stress her out"). Same recommendation, different emotional register. The voicing guide says care-centered, but the draft generator defaults to pragmatic.

- **Edit ratio 0.0 dominated this session** (8 of 11 messages sent as-is). But the 3 that were steered were the complex/sensitive ones. The system should expect: routine messages → near-zero editing, complex messages → significant steering. Don't optimize for the average — optimize for the hard cases.

## Open Questions

- How should the tool handle the transition from "investigate" to "draft"? This session did it conversationally, but a GUI needs a clear mode switch or integrated view.
- When multiple admin actions are needed (publish + push deadlines + announce), what's the right confirmation UX? One-by-one or a batch action plan the teacher reviews once?
- The topic keyword tagger was 0% accurate (everything was "other"). Should it be replaced entirely with LLM-based tagging during the scan phase, or fixed with better keyword coverage?
- How does session context decay? "I'm sick" applies for weeks. "I'm losing my job" is ongoing. The system needs to know when to stop applying old session context without being told.
