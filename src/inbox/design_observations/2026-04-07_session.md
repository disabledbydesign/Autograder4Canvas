# Session 2026-04-07 — Design Observations

## Session Stats
- Messages: 6 sent (+ 1 unanswered found via manual search), 0 skipped, 0 flagged
- Actions: 1 session_context, 7 send
- Steering instructions: ~11 total across 4 messages (0 on 3 messages sent as-is)

## What Teachers Need (generalizable patterns)

- **"What have I already said?" is a first-class workflow step.** The teacher asked to review recent announcements and mass emails before fetching the inbox. That context shaped every single draft (grade display, module confusion, point value increases, free-form late-semester mode). This should be automatic: pull announcements from the last 2 weeks and surface them before drafting begins — not as a dump, but as a parsed summary of what's already been communicated to students.

- **Instructor admin decisions need to be knowable by the system.** A student's grade display question (100% but F) required the teacher to explain a Canvas settings choice she had made (hiding the letter grade). The system drafted a generic technical explanation; the actual answer was completely different. There is a category of knowledge — "what I actually did in Canvas this week and why" — that no context file can supply. Worth exploring whether teachers can log admin decisions (module publishes, grading scheme changes, deadline pushes) somewhere the system can read.

- **Late-semester course state is its own context type.** "We're more free-form now," "point values are increasing," "the source library is open" — these aren't in the grading config or syllabus. They're decisions made in response to how the semester is actually going. Multiple drafts this session needed this info and the system didn't have it. A lightweight "current course mode" note the teacher can update would solve this.

- **Unanswered conversations aren't always in the unread queue.** One substantive message (Amber Ott) had been opened and marked read without a reply. It only surfaced because the teacher remembered it. The system should scan recently-read conversations for ones with no outgoing reply and flag them.

## System Adaptation Points

- **Burnout (0.91 edit ratio) and extra credit (0.77) are the weakest draft categories.** Burnout drafts trend too prescriptive; extra credit drafts trend too effusive. Both need calibration toward June's register: practical and brief, care built into structure not language.

- **Drafts should not validate implicit assumptions that devalue the course.** One student framed missing class as caused by prioritizing "real" STEM courses. The system validated that framing without flagging it. For a course with explicit political stakes (ethnic studies, Indigenous studies), this is a real failure. The system needs to recognize when a student message contains an assumption the instructor might want to name — not to lecture the student, but to give the instructor the choice.

- **Grouping at scan time must hold through drafting.** Messages grouped together in the dashboard were split up during drafting, which the teacher noticed and flagged. Rule: if messages are in the same group at scan time, they must be drafted together — one template shown side by side — unless the teacher explicitly says to split them.

## Missing Capability

- **Instructor identity as a context source.** A student writing about algorithmic bias prompted the teacher to share that she does her own developer research on the topic — which made the response feel personal and specific rather than generic. The system had no way to know this. More broadly: instructors aren't just course administrators. They have research areas, lived experiences, and professional identities that are directly relevant when students write about related topics. A system that only knows the syllabus and grading config is missing half the picture. The design question: how do you let an instructor register "here's who I am and what I work on" in a way the system can match against student messages without requiring them to maintain a complex profile? A lightweight tagging system — instructor-defined topic areas that trigger pulling a bio or research summary — might be enough.

## Open Questions

- How should the system represent "current course mode" — a freeform note the teacher updates? Auto-inferred from recent announcements? Both?
- Should the system proactively scan read-but-unanswered conversations at session start, or only when the teacher mentions it?
- How do you build a "flagging implicit assumptions" feature without making the system preachy or over-flagging benign messages?
- How much instructor identity context is enough? A paragraph bio? Tagged research areas? A full briefing doc? What's the minimum that actually changes draft quality?
